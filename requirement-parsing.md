# 需求文档解析

将甲方 Word/Markdown/表格转为 **requirement-map**，作为一步到位的唯一范围契约。

## 解析思想

需求解析的目的不是复述原文，而是把「能不能交付」变成可检查清单。每个功能都要落到数据、接口、页面、权限、验收和风险；落不下去的内容就是歧义或缺口。

- **先锁 P0**：登录后的主业务闭环、管理端入口、权限和数据落库优先
- **再拆边界**：哪些是后台管理，哪些是用户端/回调/定时任务，哪些明确不做
- **最后补安全性能**：越权、并发、幂等、缓存、索引、导入导出体量必须写入 requirement-map
- **默认假设可执行**：需求没说但不阻塞时，选择基座最常见做法并记录假设；阻塞时只问必要问题

## 解析规则

1. **每条 Fxx 必须映射**到：表 + 后端接口 + 前端页 + 菜单权限 — 缺一项则本轮补齐
2. **字段**以需求为准；枚举写进 dict/数据字典 SQL
3. **「类似 XXX 系统」** — 只借鉴交互，不复制无关表
4. **矛盾需求** — 列出假设，默认取对甲方最安全/最小实现
5. **工作流关键词**（审批、会签、驳回）→ 基座切 activiti/process 或内置 flow
6. **Redis 锁/TTL** → requirement-map 列 A/B/C/D（cross-cutting §3）
7. **MQ / DB Job** → **仅** requirement 写明并发峰值或长时异步时规划（§4）
8. **菜单/权限**不能只写页面路径；必须给出权限字、按钮粒度和 SQL/种子数据来源
9. **验收标准**要能手工执行：账号、入口、动作、预期结果、无权限结果
10. **文档交付**必须映射到最终代码：架构看模块，功能看页面/API，数据库 schema 看 SQL，README 看启动验收
11. **AI 边界**必须写清：本轮做什么、不做什么、哪些核心基座文件禁止改、哪些变更会影响兼容

## 与 AI_RULE 第二条对齐

解析完成后才算「阅读需求」结束；然后：

- ② 目录结构（基座 + 新业务模块）
- ③–⑨ 调用链/表/缓存/事务/权限 — 对 P0 功能逐个过一遍

## 需求仅一句话时

Agent 展开最小可行清单（登录 + 主 CRUD + 菜单 + 角色），并在 requirement-map §8 写默认假设，**仍一步到位交付**。
